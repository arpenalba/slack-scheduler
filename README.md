# Slack Status Scheduler

A personal Node.js background service that automatically changes your Slack status on a configurable schedule. Pair it with the included browser-based visual editor to manage your rules without touching JSON by hand.

---

## How it works

The service reads `config.json` once at startup and keeps it in memory. A `node-cron` job fires every 15 minutes (at `:00`, `:15`, `:30`, `:45`). On each tick it evaluates the current time against your rules and calls the Slack API only when the active rule changes — not on every tick. This means zero unnecessary API calls.

The visual editor (`editor/index.html`) is a fully static HTML file with no server dependency. It runs in your browser, lets you build and preview rules visually, and writes the updated `config.json` directly to disk (in browsers that support the File System Access API, such as Chrome and Edge). After saving, you reload the service to apply the new config.

## Getting a Slack User Token

The service needs a **User Token** (starts with `xoxp-`) with the `users.profile:write` scope. Bot tokens will not work — Slack requires a user token to set a user's own status.

1. Go to [api.slack.com/apps](https://api.slack.com/apps) and click **Create New App → From scratch**.
2. Give it a name (e.g. "Status Scheduler") and pick your workspace.
3. In the left sidebar, go to **OAuth & Permissions**.
4. Scroll to **User Token Scopes** and add `users.profile:write`.
5. Click **Install to Workspace** at the top of the page and authorize.
6. Copy the **User OAuth Token** (starts with `xoxp-`).

Open `.env` and paste the token:

```
SLACK_USER_TOKEN=xoxp-your-token-here
```

> **Never commit `.env` to version control.** It is already listed in `.gitignore`.

## Configuration

The schedule lives in `config.json` at the project root. The file has three top-level keys:

```json
{
  "timezone": "Europe/Madrid",
  "weekly": [ ... ],
  "once": [ ... ]
}
```

### `timezone`

Any valid [IANA timezone identifier](https://en.wikipedia.org/wiki/List_of_tz_database_time_zones), e.g. `"America/New_York"`, `"Europe/London"`, `"Asia/Tokyo"`. All `from`/`to` times in your rules are interpreted in this timezone.

### `weekly` — Recurring rules

Each entry applies on specific days of the week, optionally within a time window.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string (UUID v4) | yes | Unique identifier. Generated by the editor. |
| `emoji` | string | yes | Slack emoji code, e.g. `:house:` |
| `text` | string | yes | Status text shown in Slack |
| `days` | string[] | yes | Days this rule applies: `"monday"` … `"sunday"` |
| `from` | string (`HH:MM`) | no | Start of the time window (inclusive). Must be a 15-minute multiple. |
| `to` | string (`HH:MM`) | no | End of the time window (exclusive). Must be a 15-minute multiple. |

If `from` and `to` are omitted the rule applies all day on the listed days.

**Example — All-day status:**
```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "emoji": ":house:",
  "text": "Working from home",
  "days": ["monday", "friday"]
}
```

**Example — Status during a specific window:**
```json
{
  "id": "c3d4e5f6-a7b8-9012-cdef-123456789012",
  "emoji": ":fork_and_knife:",
  "text": "Lunch",
  "days": ["monday", "tuesday", "wednesday", "thursday", "friday"],
  "from": "13:00",
  "to": "14:00"
}
```

### `once` — One-time overrides

Each entry applies on a single specific date and overrides any weekly rules for that day.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | string (UUID v4) | yes | Unique identifier. Generated by the editor. |
| `emoji` | string | yes | Slack emoji code |
| `text` | string | yes | Status text |
| `date` | string (`YYYY-MM-DD`) | yes | The specific date this applies |
| `from` | string (`HH:MM`) | no | Start of the time window (inclusive) |
| `to` | string (`HH:MM`) | no | End of the time window (exclusive) |

**Example — PTO for an entire day:**
```json
{
  "id": "d4e5f6a7-b8c9-0123-defa-234567890123",
  "emoji": ":beach_with_umbrella:",
  "text": "PTO",
  "date": "2026-05-15"
}
```

### Priority order

When multiple rules could apply at the same time, the service picks the first match in this order (highest to lowest priority):

1. `once` rule **with** a time window — for the current time slot
2. `once` rule **all-day** — for today's date
3. `weekly` rule **with** a time window — for the current time slot
4. `weekly` rule **all-day** — for today's day of week
5. No match → Slack status is cleared

Within the same priority level, the **first rule in the array wins**.

### Time rounding

All times are floored to the nearest 15-minute boundary before comparison. If the cron fires at `13:47`, it is treated as `13:45`. This means only `00`, `15`, `30`, and `45` are valid minute values for `from` and `to`. The editor enforces this automatically.

### Time window semantics

- `from` is **inclusive**: at `13:00` the status activates.
- `to` is **exclusive**: at `14:00` the status is already gone.
- Ranges **cannot cross midnight** — `22:00` to `02:00` is invalid.

## Running the service

### Option A — Interactive mode (development)

```bash
node scheduler/index.js
```

The service starts, evaluates the current time immediately, and presents a console menu:

```
1) Open editor  2) Reload config  3) Quit
>
```

| Option | What it does |
|---|---|
| `1` | Opens `editor/index.html` in your default browser |
| `2` | Re-reads `config.json` from disk and applies rules immediately |
| `3` | Exits the process |

### Option B — Background service with PM2 (recommended for daily use)

[PM2](https://pm2.keymetrics.io/) keeps the service running after you close your terminal and restarts it on system boot.

```bash
# Install PM2 globally (one time)
npm install -g pm2

# Start the service
pm2 start scheduler/index.js --name slack-scheduler

# Enable auto-start on system boot (follow the instructions PM2 prints)
pm2 startup
pm2 save

# Check status
pm2 status

# View live logs
pm2 logs slack-scheduler

# Stop the service
pm2 stop slack-scheduler

# Apply config changes (reload the process)
pm2 restart slack-scheduler
```

## Using the visual editor

Open `editor/index.html` directly in your browser (double-click it in Explorer, or use option `1` from the console menu). No web server is required.

### Loading your config

Click the folder icon (📁) at the top of the **Config** section and pick your `config.json`. In Chrome and Edge (File System Access API supported), the editor remembers the file between sessions and can save directly back to disk. In other browsers, you download the updated file and replace `config.json` manually.

### Weekly rules

- Click **+ Add rule** to create a new weekly rule.
- Click the day pills (L M X J V S D) to toggle which days the rule applies.
- Enter the emoji code in `:colon-format:` (e.g. `:house:`, `:office:`). Standard Unicode emoji are not accepted here — Slack requires the colon format.
- Enable **Time range** to restrict the rule to a specific window. Times snap to 15-minute intervals.
- Rules are applied in list order — drag or re-add rules to change priority.

### Once entries

- Click **+ Add entry** to create a one-time override.
- Pick the date using the date picker. Entries in the past are shown faded with an **Expired** badge — they are harmless to keep but can be deleted.

### Preview Simulator

Use the day pills and time selector in the **Preview Simulator** to test which status would be active at any given day and time. This runs entirely in the browser against the current in-memory rules — no API calls are made.

### Saving

| Button | Behaviour |
|---|---|
| **Save** | Writes `config.json` directly to disk (Chrome/Edge only). Disabled if any rule is invalid. |
| **Reload** | Re-reads the file from disk (useful after external edits). |
| **Copy JSON** | Copies the current config JSON to the clipboard. |
| **Export** | Downloads `config.json` to your Downloads folder (fallback for all browsers). |

Cards with a **red border** are invalid (missing emoji, missing text, no days selected, or `from` ≥ `to`). Save and Export are disabled until all rules are valid.

## Applying config changes

| Mode | How to reload |
|---|---|
| Interactive (`node scheduler/index.js`) | Press `2` at the console menu |
| PM2 | Run `pm2 restart slack-scheduler` |

The service never reads `config.json` automatically after startup — the reload is always explicit. This avoids partial reads during a file download or save.

## Logs

The service appends a line to `logs/status.log` every time the Slack status changes:

```
2026-05-11T09:00:00.000Z SET emoji=:house: text=WFH httpStatus=200
2026-05-11T13:00:00.000Z SET emoji=:fork_and_knife: text=Lunch httpStatus=200
2026-05-11T14:00:00.000Z CLEAR emoji= text= httpStatus=200
```

The log is capped at 500 lines; older entries are trimmed automatically.


## Limitations

- **No midnight-crossing ranges.** A rule like `22:00 → 02:00` is not supported and will produce incorrect results. Use two separate rules instead.
- **15-minute resolution.** The scheduler only fires at 15-minute boundaries. Sub-15-minute precision is not possible.
- **Single workspace.** The token is tied to one Slack account in one workspace.
- **No automatic config file watching.** The service must be explicitly restarted or reloaded after editing `config.json`.
